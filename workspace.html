Design Concept — Speculative UI/UX

The Agentic Workspace

A prototype layout for how the Product Engineer observes agents and reviews the artifacts that require human judgment — designed around the principle that the PE’s primary interface is a prioritized decision queue, not a log viewer or project board.

The Core UX Insight

Most agent monitoring UIs are built for engineers debugging systems. This workspace is built for a Product Engineer making product decisions. The distinction is critical: the interface surfaces what needs a judgment call, not what the agents are doing internally. Agent activity is context; artifact review is the job. Everything in this layout is organized around that hierarchy.

Scenario

A neutral, representative scenario: a B2B SaaS analytics platform adding a new capability mid-lifecycle. The workspace below shows a snapshot during Engineering Sprint 2 of 3, with some agents complete, one blocked, and several critical artifacts awaiting review.

Three modes · one workspace

Focus · Flow · Map

The Product Engineer switches between three cognitive modes through the day: deciding, tuning in, and steering. One layout can’t serve all three — so the workspace offers three, each stripped to what that mode actually needs.

Focus — the Decision Inbox

One decision fills the viewport. No roster, no feed, no escalations panel — just the next judgment call, fully packaged. Keyboard-driven (J/K to move, A approve, E escalate, S skip). Use this mode when the queue has backed up and you need to clear it without distraction. Each card surfaces agent confidence and 30-day recommendation accuracy, so you can calibrate trust against the actual agent, not a generic AI.

Flow — the Ambient Stream

A live vertical stream: agents working up top with pause / redirect controls, time-ordered events below, decisions appearing inline as they arise. No separation between “what agents are doing” and “what needs me” — they’re the same stream, filtered. Use this mode when you want to stay in the loop without blocking on every artifact. The “Now working” bar makes the missing agentic affordance — interrupt mid-task — a first-class control.

Map — the Command View

The three-column roster + queue + escalations layout. Use this mode for situational awareness: where is the feature, who’s blocked, what’s the confidence shape across the four dimensions, where’s the sprint relative to plan. The Map is for steering decisions (reallocate, re-scope, unblock) rather than reviewing artifacts. Agent items now carry a trust signal (recommendation acceptance rate) and the active ones expose a pause control.

Each mode keeps what’s load-bearing for a specific task, drops the rest. That’s the opposite of “a dashboard with everything on it” — which is what most agent UIs still are.

UX / Design Rationale

Why This Layout

Key decisions behind the workspace structure — each one is a deliberate UX position, not a default.

Artifact queue is the dominant element

The center column (the widest) holds only the review queue. Agent status and activity are secondary, in narrower columns. This forces the layout itself to communicate the PE’s actual job: making decisions, not monitoring dashboards. The queue is never just a list — each item is a pre-packaged decision with agent reasoning, options, and a clear call to action.

Tier system enforces review discipline

Tier 1 (deep), Tier 2 (spot check), Tier 3 (scan) are visually separated with clear time budgets shown in the headers. This prevents the PE from spending 30 minutes reviewing a changelog draft or skimming a legal finding. The tier is set by the agent based on artifact type — the PE doesn’t categorize, they process.

Escalations are separate from the queue

Escalations require a different cognitive mode than artifact review: they are conflicts or blocks, not decisions packaged for approval. Showing them in the right panel (visible but not dominant) lets the PE hold them in peripheral attention while working through the queue. The “Review artifact #01” button links escalations back to the relevant artifact, keeping the queue as the single point of action.

Agent roster shows state, not activity

The agent roster doesn’t show what agents are currently executing — it shows their lifecycle state (complete, in progress, blocked, standby). The activity feed handles the narrative of what happened. The roster is for situational awareness at a glance: which agents are done, which are running, which need PE input. Confidence scores appear only where a score exists and is meaningful.

Blocking state surfaces immediately

The “Blocked” badge on the Legal Agent, the red summary stat, and the escalation panel all refer to the same issue from three different vantage points. This redundancy is intentional: a sprint-blocking issue should be unmissable regardless of where on the page the PE is looking. The queue intro also calls it out explicitly in plain language.

PDLC stage bar provides context, not control

The stage indicator at the top shows where this feature is in the lifecycle, giving the PE the frame for interpreting agent behavior. It’s not a navigation tool — the PE doesn’t click it to move stages. Stage transitions happen when confidence thresholds and gates are met, not by clicking a button. The current sprint counter adds temporal grounding without requiring a separate calendar view.

Feature readiness is a single derived number

The 61% readiness score is the Orchestrator’s aggregate view across the four confidence dimensions (value, usability, feasibility, viability), weighted by lifecycle stage and unblocked by open items. It gives the PE a single answer to “how far are we?” without requiring them to mentally average four scores. It also degrades visibly when blockers exist — the 61% reflects both confidence and blockage.

Artifacts bundle context with the decision

Each artifact card includes: who raised it, how long ago, what the agent found, what the options are, the agent’s recommendation (labeled as such, not presented as fact), and clear action buttons. The PE should never need to open a separate tool, read a thread, or ask a follow-up question to make the decision. If the context is insufficient, the “Add context note” / “Request more info” actions let the PE send feedback back into the agent loop.
