Proposed additional categories draft · for discussion

The three-layer rail (ambient · agentic · ephemeral) plus the cross-cutting bin has held up for five weeks — but several themes recur often enough that they probably deserve their own lens. The Graph view already toggles colour-by-vendor and colour-by-theme; these would slot in as additional filter dimensions, not replacements for the layer rails.

  1. Maturity stage. Research · Preview · GA · Procurement · Deployment scale. Currently a paper, a launch, and a $30bn raise all collapse into one feed; separating them shows which layer is in which phase.
  2. Vertical. Healthcare, Legal, Life sciences, Finance, Supply chain, DevSecOps, Public sector. Vertical tags already exist ad-hoc — promoting them to a dimension makes “agents are real first in regulated industries” a measurable claim, not an anecdote.
  3. Autonomy level. Borrowed from the SAE driving taxonomy: L1 assistive · L2 supervised · L3 conditional · L4 high · L5 full. Lets us track where on the autonomy curve each release sits, and where trust-calibration UX is concentrating.
  4. PDLC stage. Discover · Define · Build · Verify · Ship · Operate. Most current signals are Build/Verify; the under-served stages would surface immediately.
  5. Standard vs. product. Splitting Standard from Product launch from Research from Capital event from Procurement. A protocol release (A2UI, MCP) belongs in a different lens than a product launch (Antigravity 2.0) than an alliance (KPMG × Anthropic).
  6. Time-to-impact. This quarter · This year · Multi-year. Aligns the log with the three-horizon framing the writeup already uses.
  7. Workforce shape. Tooling change · Role change · Org change · Procurement change. The most strategically interesting signals (e.g. KPMG × Anthropic, ServiceNow + Accenture FDE) are role/org changes; today they get filed as “cross-cutting” alongside funding rounds.

Implementation note. These would go onto each <article class="sig-item"> as data-* attributes (e.g. data-maturity="GA" data-vertical="legal" data-autonomy="L3") so the existing tag chips stay clean and the Graph view can offer them as additional “colour by” / “filter by” axes without restructuring the page. Recommend piloting Maturity and Vertical first — both are visible in the data already and would change which signals get prioritised in any given week.

How this page is produced

Each Sunday evening, the refresh-signals Claude Code skill (in .claude/skills/refresh-signals/) runs the same scan we ran for the seed entries:

  1. Scope. Last 7–14 days only. Strictly enforced — older items belong in the writeup, not here.
  2. Topics. Agentic PDLC, the three layers (ambient / agentic / ephemeral), product engineer role, harness engineering, trust calibration, multi-agent orchestration, generative UI standards.
  3. Per signal. 2–5 sentences. One source link in the title. Plain English. Why-it-matters first, what-it-is second.
  4. Layering. Each item is filed under one of three layers, or the cross-cutting rail (orchestration patterns, role evolution, regulation). Five items per layer per week is plenty — signal beats volume.
  5. Append, don’t overwrite. Each new week is added as a new <section class="sig-week"> at the top; older weeks stay below for longitudinal reading.

The skill produces a draft block that the PE reviews before commit — this is a Layer 3 surface in its own right. To run it: invoke the skill from Claude Code on this repo and follow the prompts.
